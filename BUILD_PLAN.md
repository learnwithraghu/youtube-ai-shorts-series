# Plan: Build the Shipping Support Assistant + DeepEval Harness (Multi-Session Code Build)

## Context

`deep-eval-course` currently contains only course documentation (`sections/`, `skills/`, `COURSE_*.md`) — zero code, zero config, one git commit. The course outline (`COURSE_OUTLINE.md`) already specifies a target repo layout (`app/`, `tools/`, `data/`, `evals/`, `reports/`, `docs/`) and a "shipping support assistant" example used throughout. The goal now is to build the **actual runnable code** behind the lessons — the assistant, its mock backend, the DeepEval harness, the regression workflow, and the reporting layer — so each course section has a real, working artifact to teach from, not just a description.

Because this is a large build, it's broken into discrete sessions, each independently completable in one sitting, each ending in a concrete "done check." Sessions map onto course sections so the code and lessons stay in lockstep, but the code itself lives outside `sections/**` (this plan does not edit any `sections/**/README.md` files).

## Decisions agreed with user

| Area | Decision |
|---|---|
| LLM provider (assistant + judge) | OpenAI for both |
| Dependency manager | `uv` |
| Demo UI | Streamlit app |
| Mock backend data | JSON fixture files under `data/backend/` |
| Tracing/observability | LangSmith (auto-instrumented tracing; still teach trace/run structure conceptually, and export a local JSONL copy of key trace metadata so failure-capture/regression tooling doesn't require LangSmith API access) |
| Eval test execution | Pytest-style DeepEval tests (`deepeval test run` / `pytest evals/`) |
| Regression baselines | Versioned JSON files (`evals/regression/baselines/<version>.json`) |
| CI LLM calls | Cached/recorded responses replayed in CI; live runs are a manual/local step |

Defaults adopted without a separate question (low-stakes / clearly implied by existing docs):
- CI provider: **GitHub Actions** (repo is on GitHub).
- Orchestration: **plain LangChain routing** for simple intents, **LangGraph `StateGraph`** specifically for the stateful/multi-turn parts (clarification loops, escalation) — matches chapter 02-03's "when LangGraph-style stateful orchestration helps."
- Metric implementation: DeepEval built-ins (`FaithfulnessMetric`, etc.) for correctness/groundedness; custom `GEval` rubrics for tone/safety; custom `BaseMetric` for tool-use correctness (no built-in covers it).
- Reporting format: Markdown report files generated by script, optionally also surfaced as a tab in the Streamlit app (since Streamlit is already in the stack).

## Verification approach (overall)

- Each session's "done check" above is the local acceptance test for that session — run the listed command(s) before moving to the next session.
- From Session 9 onward, `pytest evals/` is the standing regression suite; it should stay green (or show only intentional, explained failures) at the end of every subsequent session.
- Session 14 is the end-to-end verification that the whole chain (app → traces → evals → thresholds) works without manual intervention, gated in GitHub Actions.
- Final verification (end of Session 17): clone-and-follow-README test described above, run by a fresh `uv` environment.
